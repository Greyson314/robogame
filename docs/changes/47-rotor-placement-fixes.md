# 47 — Rotor placement fixes (FP overlap, spin-axis face, hidden cube)

> Three user-reported regressions on the rotor / blade placement
> path, traced and fixed.

## What changed

### 1. Swept-volume overlap false positive on side-mounted blades

**Symptom (user):** Place a default-span blade on the left or right
face of the helicopter mechanism cube → "Swept Volume Overlaps a
Neighbor". Same trigger when a perpendicular blade already exists.

**Root cause:** [`BlockOccupancy.ComputeFoilSweptBoundsLocal`](../../Assets/_Project/Scripts/Block/BlockOccupancy.cs)
rotates the foil's local half-extents through `BlockGrid.OrientationFromUp(up)`'s
quaternion to get chassis-axis-aligned bounds. Quaternion-vector
multiplication accumulates ~1e-7 floating-point error per axis, so a
span-1 foil at (1,3,0) up=+X comes out with `bounds.min.x ≈ 0.5 - 1e-7`
instead of exactly 0.5. The mechanism cube's `bounds.max.x` is
exactly 0.5, and `StrictOverlap`'s `amin.x < bmax.x` comparison flagged
the FP-shifted edge-touching as real overlap. Up=+Y placements were
unaffected because identity rotation is exact.

**Fix:** Add a 1e-3 epsilon on each axis in
[`BlockOccupancy.StrictOverlap`](../../Assets/_Project/Scripts/Block/BlockOccupancy.cs).
Real interpenetration crosses cell boundaries by orders of magnitude
more, so the epsilon doesn't mask genuine overlaps (regression test
covers span-2 case).

### 2. Mechanism cube visible in the garage

**Symptom (user):** The default helicopter's invisible mechanism cube
*isn't* invisible in the garage — it shows on top of the rotor stem,
visually clipping the rotor's disc and crossbars.

**Root cause:** [`RotorBlock.RefreshMechanismCellMeshVisibility`](../../Assets/_Project/Scripts/Movement/RotorBlock.cs)
only hides the cube when the chassis is non-kinematic (i.e. in an
arena). The original rationale was "show the cube in garage so a
player who placed a rotor under an existing block doesn't see that
block disappear" — but the cost is a much louder UX problem: the
rotor's blade ring height (cell-y centre + 1.0) is exactly where the
mechanism cube sits, so disc + bars + cube all stack at the same
spot.

**Fix:** Hide the mechanism cube's mesh always. The cube stays in the
grid (collider, damage routing, connectivity intact) — only its
renderer is suppressed. Future enhancement: distinguish auto-placed
mechanism cubes (always hide) from user-placed structural cubes that
happen to be at a rotor's mechanism cell (show those).

### 3. Nothing places on a rotor in from-scratch builds

**Symptom (user):** Drop a fresh rotor in build mode, try to place
*anything* on it (mechanism cube, blade, whatever) → rejected.

**Root cause:** The rotor is in `BlockConnectivity.s_hardcodedLeafIds`,
which means *no* face accepts a host. That's correct for the
lateral faces (you don't mount a wing on a rotor's side) but wrong
for the spin-axis face (the mechanism cube is supposed to mount
there). The default helicopter preset works because it ships the
mechanism cube pre-placed; from-scratch builds can't reproduce that
layout.

**Fix:** New
[`BlockConnectivity.IsConnectiveFace(hostDef, hostUp, placementUp)`](../../Assets/_Project/Scripts/Block/BlockConnectivity.cs)
defaults to "non-leaves accept all, leaves accept none" but adds a
rotor exception: a rotor's spin-axis face (its `Up` direction) is
connective. Used by both
[`PlacementRules.CheckHostIsConnective`](../../Assets/_Project/Scripts/Block/PlacementRules.cs)
and the [`BlueprintValidator`](../../Assets/_Project/Scripts/Block/BlueprintValidator.cs)
per-entry leaf-host pass.

Also fixed [`BlueprintBuilder.RotorWithFoils`](../../Assets/_Project/Scripts/Block/BlueprintBuilder.cs)
to author the mechanism cube + each blade with their proper mount-up
vectors (cube up = spin axis; blade up = its outward direction from
the cube), so blueprints generated by the scaffolder now describe a
build a user could actually reproduce manually. Existing helicopter
preset asset on disk still has the old per-foil up=+Y; re-running the
scaffolder ("Robogame → Build Everything") refreshes it. The runtime
still adopts old-up foils correctly because adoption overrides
transform rotation regardless of `b.Up`.

## On the user's "old tech" worry

The user thought the rotor was "created with 4 blades inseparably
tied to it" based on console logs. Not the case: `[RotorBlock]
'name': BuildLiftRig adopted N foil(s)` is the only relevant log,
and "adopted" is the correct verb — the four blades are *separate*
`BlockBehaviour` entries that the rotor reparents to its kinematic
hub at chassis spawn. Removing one with right-click cleanly destroys
just that blade GameObject. The rotor's `_adoptedFoils` list ends up
holding a stale null reference, but the FixedUpdate path doesn't
iterate the list (the hub spins as a unit), so it's harmless.

   blades on every face of the mechanism cube. All four should accept.
2. Same chassis, default helicopter preset → look at the rotor head
   (top of the chassis). The disc + crossbars should be the only
